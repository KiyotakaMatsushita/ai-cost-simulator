# AI コストシミュレーター 計算ロジック説明書

## 概要

本シミュレーターは、Gemini（2.5 Flash / 3 Flash / 3 Pro）、Claude（Haiku 4.5 / Sonnet 4.5 / Opus 4.5）、OpenAI（GPT-5 Mini / GPT-5）の8モデルにおいて、**コンテキストキャッシュを利用した場合**の運用コストを比較するツールです。

---

## 入力パラメーター

| 記号 | パラメーター名 | デフォルト値 | 説明 |
|------|---------------|-------------|------|
| C | 初期キャッシュ量 | 50,000 tokens | キャッシュに書き込むトークン数（システムプロンプト等） |
| R | 再利用ヒット量 | 50,000 tokens | 1リクエストあたりのキャッシュ読み出しトークン数 |
| P | 新規プロンプト | 7,000 tokens | 1リクエストあたりの新規入力トークン数 |
| O | 回答トークン | 2,000 tokens | 1リクエストあたりの出力トークン数 |
| U | 利用者数 | 2,500 人 | テスト対象のユーザー数 |
| N | 回数/ユーザー | 10 回 | 1ユーザーあたりのリクエスト回数 |
| D | テスト日数 | 7 日 | 運用期間 |
| h | Gemini保持時間 | 24 時間/日 | 1日あたりのキャッシュ保持時間（Geminiの保管料計算用） |

### 導出値

| 値 | 計算式 | 説明 |
|----|--------|------|
| 総リクエスト数 | `U × N` | 全ユーザーの合計リクエスト数 |
| 総保管時間 | `h × D` | キャッシュの累計保管時間 |
| スケール倍率 | 共有: `1` / 個別: `U` | キャッシュが共有か個別かで変わる |

### 課金モデルの違い

- **Gemini**: 書き込みは入力と同額（上乗せなし）。代わりに保管料が時間単位で発生。永続キャッシュ（明示削除まで保持）。
- **Claude**: 書き込みに上乗せ（5分TTL: 1.25倍、1時間TTL: 2.0倍）。保管料は無料。TTL内にリクエストがあればキャッシュが自動延長される（sliding window）。
- **OpenAI**: 書き込みは入力と同額（上乗せなし）。保管料も無料。自動キャッシュ（約5分TTL）で、明示的なキャッシュ作成は不要。50%割引のcached input価格が適用される。

## 計算式（5段階）

### 1. キャッシュ作成費（Write Cost）

キャッシュを作成する際のコスト。書き込み回数/日は手動指定または自動計算。

```
Write Cost = (C / 1,000,000) × WritePrice × WritesPerDay × D × ScaleMultiplier
```

- **Gemini の WritePrice** = Input単価（上乗せなし）、WritesPerDay = 1（永続キャッシュ）
- **Claude の WritePrice** = 5分TTL: Input × 1.25 / 1時間TTL: Input × 2.0
- **OpenAI の WritePrice** = Input単価（上乗せなし）、自動キャッシュ（~5分TTL）
- **ScaleMultiplier** = 共有キャッシュ: 1 / 個別キャッシュ: U

#### Claude/OpenAI書込回数/日の自動計算（Sliding Window TTL）

Claude・OpenAIのキャッシュはsliding window方式で、TTL期間内にアクセスがあるとTTLがリセットされます。リクエスト頻度がTTLより短い場合、キャッシュは失効せず再書き込みは不要です。（OpenAIは固定~5分TTL、Claudeは5分/1時間TTLを選択可能）

```
リクエスト間隔 = 稼働時間/日 ÷ リクエスト数/日/キャッシュ
  - 共有: リクエスト数/日 = (U × N) ÷ D
  - 個別: リクエスト数/日 = N ÷ D

if リクエスト間隔 ≤ TTL:
  WritesPerDay = 1  （キャッシュ維持 — 初回書き込みのみ）
else:
  WritesPerDay = ceil(リクエスト数/日)  （毎回再書き込み）
```

**例: Gemini 2.5 Flash（共有キャッシュ、WritesPerDay=1）**
```
= (50,000 / 1,000,000) × $0.30 × 1 × 7 × 1
= 0.05 × 0.30 × 7
= $0.105
```

**例: Claude Haiku 4.5（共有キャッシュ、1h TTL、自動=1回/日）**
リクエスト間隔 = 24h ÷ (25,000 ÷ 7) ≈ 0.007h（TTL 1h以内）→ WritesPerDay = 1
```
= (50,000 / 1,000,000) × $2.00 × 1 × 7 × 1
= 0.05 × 2.00 × 7
= $0.70
```

### 2. 保管料（Storage Cost）

キャッシュを保持している時間に対する課金。**Geminiのみ発生、Claude/OpenAIは無料。**

```
Storage Cost = (C / 1,000,000) × StoragePrice × (h × D) × ScaleMultiplier
```

**例: Gemini 2.5 Flash（共有キャッシュ、24h/日）**
```
= (50,000 / 1,000,000) × $1.00 × (24 × 7) × 1
= 0.05 × 1.00 × 168
= $8.40
```

### 3. キャッシュ再利用費（Read Cost）

各リクエストでキャッシュから読み出す際のコスト。入力単価の10%で大幅割引。

```
Read Cost = (R / 1,000,000) × CachedInputPrice × (U × N)
```

> ※ ScaleMultiplierは不要。各リクエストが1回ずつ読むため、総リクエスト数で計算。

**例: Gemini 2.5 Flash**
```
= (50,000 / 1,000,000) × $0.03 × (2,500 × 10)
= 0.05 × 0.03 × 25,000
= $37.50
```

### 4. 新規入力費（Prompt Cost）

キャッシュに含まれない新規プロンプト部分のコスト。通常入力単価で課金。

```
Prompt Cost = (P / 1,000,000) × InputPrice × (U × N)
```

**例: Gemini 2.5 Flash**
```
= (7,000 / 1,000,000) × $0.30 × 25,000
= 0.007 × 0.30 × 25,000
= $52.50
```

### 5. 出力費（Output Cost）

モデルの生成した回答トークンに対するコスト。キャッシュの有無に関係なく固定。

```
Output Cost = (O / 1,000,000) × OutputPrice × (U × N)
```

**例: Gemini 2.5 Flash**
```
= (2,000 / 1,000,000) × $2.50 × 25,000
= 0.002 × 2.50 × 25,000
= $125.00
```

## 合計コスト

```
Total Cost = Write Cost + Storage Cost + Read Cost + Prompt Cost + Output Cost
```

**例: Gemini 2.5 Flash（共有キャッシュ、24h保持、デフォルトパラメーター）**
```
= $0.105 + $8.40 + $37.50 + $52.50 + $125.00
= $223.505
```

### 1リクエストあたりのコスト

```
Per Request Cost = Total Cost / (U × N)
```

```
= $223.505 / 25,000
= $0.00894 ≈ ¥1.39（@¥155/USD）
```

## キャッシュなしとの比較（削減率の計算）

キャッシュを使わない場合、キャッシュ読み出し分（R）もすべて通常入力単価で課金されます。

```
No-Cache Cost = ((R + P) / 1,000,000) × InputPrice × (U × N) + Output Cost
```

```
削減額 = No-Cache Cost - Total Cost
削減率 = (削減額 / No-Cache Cost) × 100
```

**例: Gemini 2.5 Flash**
```
No-Cache Cost = ((50,000 + 7,000) / 1,000,000) × $0.30 × 25,000 + $125.00
             = 0.057 × 0.30 × 25,000 + 125.00
             = $427.50 + $125.00
             = $552.50

削減額 = $552.50 - $223.505 = $328.995
削減率 = ($328.995 / $552.50) × 100 = 59.5%
```

## 共有キャッシュ vs 個別キャッシュ

### 共有キャッシュ（Shared）
- 全ユーザーで1つのキャッシュを共有（例：会社のマニュアル、システムプロンプト）
- ScaleMultiplier = **1**
- Write Cost と Storage Cost はユーザー数に依存しない
- **Gemini有利**: 保管料が固定のため、ユーザー数が多いほど1人あたりのコストが下がる

### 個別キャッシュ（Personalized）
- ユーザーごとに専用キャッシュを作成（例：個人の会話履歴、専用ドキュメント）
- ScaleMultiplier = **U（ユーザー数）**
- Write Cost と Storage Cost がユーザー数に比例して増加
- **Claude有利**: 保管料が無料のため、個別キャッシュが大量にあってもStorageコストが発生しない

## まとめ

| 観点 | Gemini | Claude | OpenAI (GPT-5) |
|------|--------|--------|----------------|
| 書き込み | 入力と同額（安い） | 1.25〜2.0倍の上乗せ | 入力と同額（安い） |
| 保管料 | 時間課金（長時間保持で高額化） | 無料 | 無料 |
| 再利用 | 入力の10%（安い） | 入力の10%（同率だが基本単価が高い） | 入力の10%（50%割引） |
| TTL | 永続キャッシュ（明示的に削除するまで保持） | Sliding window（アクセスで延長、5分/1時間TTL） | 自動キャッシュ（~5分TTL、sliding window） |
| 書込回数 | 1回/日（固定） | リクエスト頻度に依存（高頻度なら1回/日で維持可能） | リクエスト頻度に依存（自動管理） |
| 共有キャッシュ | 非常に有利 | 普通 | 有利（保管料無料） |
| 個別キャッシュ | 保管料がU倍で高額化 | 保管料無料で有利 | 保管料無料で有利 |

**結論**: ユースケース（共有 or 個別）とキャッシュ保持時間によって最適なモデルが変わります。シミュレーターで各パラメーターを変更して比較してください。

## AgentCore Memory 計算ロジック

### 概要

AgentCore Memory タブでは、AWS Bedrock AgentCore Memory の月額コストを試算します。LLM の推論コストとは独立した料金体系（イベント単位・レコード単位・リクエスト単位）で、モデルに依存しません。

### 入力パラメーター

| 記号 | パラメーター名 | デフォルト値 | 説明 |
|------|---------------|-------------|------|
| U | ユーザー数 | 10,000 人 | 月間アクティブユーザー数 |
| S | セッション/ユーザー/月 | 20 回 | 1ユーザーあたりの月間セッション数 |
| RT | ラウンドトリップ/セッション | 2 回 | 1セッションあたりのユーザー⇔エージェント往復回数 |
| RV | LTM検索/セッション | 1 回 | 1セッションあたりのLTMセマンティック検索回数 |
| RC | LTMレコード/ユーザー | 10 件 | 1ユーザーあたりの蓄積LTMレコード数 |

### AWS AgentCore Memory 単価（2026年2月時点）

| サービス | 単価 | 単位 |
|---------|------|------|
| STM イベント作成 | $0.25 | /1,000 events |
| LTM ストレージ (Default) | $0.75 | /1,000 records/月 |
| LTM ストレージ (Override) | $0.25 | /1,000 records/月 |
| LTM 検索 | $0.50 | /1,000 requests |

### 導出値

| 値 | 計算式 | 説明 |
|----|--------|------|
| 総セッション数 | `U × S` | 月間の全セッション数 |
| 総STMイベント数 | `U × S × RT × 2` | 各ラウンドトリップでユーザー発話+アシスタント応答の2イベント |
| 総LTM検索数 | `U × S × RV` | 月間のLTMセマンティック検索リクエスト数 |
| 総LTMレコード数 | `U × RC` | 保存中のLTMレコード総数 |

### 計算式

#### 1. STM イベントコスト

```
STM Cost = 総STMイベント数 × ($0.25 / 1,000)
```

**例: 10,000ユーザー × 20セッション × 2RT × 2**
```
= 800,000 イベント × $0.00025
= $200.00
```

#### 2. LTM ストレージコスト

```
LTM Storage Cost = 総LTMレコード数 × (Strategy別単価 / 1,000)
```

- **Default Strategy**: `$0.75 / 1,000 records/月`
- **Override Strategy**: `$0.25 / 1,000 records/月`（67%割引）

**例: 10,000ユーザー × 10レコード = 100,000レコード**
```
Default:  100,000 × $0.00075 = $75.00
Override: 100,000 × $0.00025 = $25.00
```

#### 3. LTM 検索コスト

```
LTM Retrieval Cost = 総LTM検索数 × ($0.50 / 1,000)
```

**例: 10,000ユーザー × 20セッション × 1検索**
```
= 200,000 × $0.0005
= $100.00
```

#### 4. 合計

```
Total Memory Cost = STM Cost + LTM Storage Cost + LTM Retrieval Cost
```

**例（Default Strategy）:**
```
= $200.00 + $75.00 + $100.00
= $375.00/月
```

### 単価指標

```
セッション単価   = Total / 総セッション数
ユーザー月額     = Total / U
ユーザー年額     = ユーザー月額 × 12
```

### 基本タブとの統合

基本シミュレータータブのコスト内訳テーブルでは、Memory コスト（月額）を LLM コストに加算して表示します。

```
合計 (LLM + Memory) = LLM Total + Memory Total
1リクエスト単価     = (LLM Total + Memory Total) / 総リクエスト数
```

Memory コストは月額固定なので、LLM の総リクエスト数で按分して1リクエストあたりのコストに換算します。
